# 词库体系设计（面向 librime）

## 1. 设计原则

- 基础词库本地可用，并以 Rime schema/dict 形式组织。
- 用户词库本地优先，复用 `librime` 的学习与 `user_data` 机制。
- 行业词库和热词库采用可替换的 Rime 字典补丁包。
- 词库分层清晰，便于 deploy、灰度、回滚和 license 管理。

## 2. 词库分层

| 层级 | 用途 | 是否离线 |
| --- | --- | --- |
| `rime_schema` | Rime 方案定义，描述输入行为 | 是 |
| `core_pinyin` | 中文基础拼音词条 | 是 |
| `core_phrase` | 高频短语与长词 | 是 |
| `core_english` | 英文高频词与短语 | 是 |
| `emoji_map` | 文本到 emoji 映射 | 是 |
| `asr_bias` | 语音识别热词 | 是 |
| `domain_pack` | 行业专有词 | 可选 |
| `hot_update` | 热门新词 | 否 |
| `user_dictionary` | 用户自定义与学习结果 | 是 |
| `blocklist` | 敏感、误触、错误词条拦截 | 是 |

## 3. Rime 资源组织

采用 `librime` 后，词库不只是“数据文件”，而是“方案 + 字典 + patch + deploy 产物”的整体。

建议目录：

```text
rime/shared/
  default.yaml
  offhand_pinyin.schema.yaml
  offhand.base.dict.yaml
  offhand.domain.office.dict.yaml
  offhand.hotword.dict.yaml
  offhand_pinyin.custom.yaml
  build/
    offhand_pinyin.prism.bin
    offhand.base.table.bin
    offhand.base.reverse.bin
```

用户目录：

```text
<app_private>/rime/
  build/
  user.yaml
  offhand_pinyin.userdb/
  custom_phrase.txt
```

## 4. 官方包与 license 策略

建议优先参考或复用官方 Rime 生态中的以下项目：

| 组件 | 建议用途 | 备注 |
| --- | --- | --- |
| `librime` | 输入法核心库 | 核心库为 BSD-3-Clause |
| `rime-luna-pinyin` | 全拼方案参考或基础继承 | 官方方案仓库，GitHub 显示为 LGPL-3.0 |
| `rime-essay` | 基础共享词汇与语言模型参考 | 官方词汇仓库，GitHub 显示为 LGPL-3.0 |
| `plum` | 构建期 recipe/配置管理参考 | 只建议用于离线构建流程，不建议作为客户端运行时依赖 |

落地建议：

- 技术上可先参考 `luna_pinyin` 的 schema 结构快速起步。
- 商业发布前要做一次 license 审核，明确是否直接分发第三方方案包，还是转为自维护的 `offhand_*` 方案与词库。
- 首版客户端运行时不依赖 `plum`，它只保留在数据打包或离线构建流水线中。

## 5. 中文词库源格式建议

建议把 TSV/CSV/自有词库表作为“源数据”，再转换为 Rime `*.dict.yaml`。

源数据字段建议如下：

| 字段 | 说明 |
| --- | --- |
| `term` | 词条文本 |
| `pinyin` | 全拼，音节间空格分隔 |
| `pinyin_abbr` | 简拼 |
| `weight` | 基础权重 |
| `pos` | 词性 |
| `domain` | 领域 |
| `flags` | 标签，如 `common`、`spoken`、`person_name` |
| `source` | 来源 |

示例见：

- `design/samples/base-pinyin-lexicon.sample.tsv`

## 7. Rime 字典拆分建议

建议不要把所有中文词条塞进单一大字典，而是拆成可治理的层次：

| 字典文件 | 内容 |
| --- | --- |
| `offhand.base.dict.yaml` | 通用高频基础词 |
| `offhand.phrase.dict.yaml` | 高频短语与固定搭配 |
| `offhand.domain.<name>.dict.yaml` | 行业词包 |
| `offhand.hotword.dict.yaml` | 动态热词 |
| `offhand.blocklist.dict.yaml` | 排除或降权控制词 |

策略：

- 基础包随应用发版。
- 行业包与热词包可灰度更新。
- Blocklist 必须支持远程快速下发。

## 8. ASR 热词格式

用于提升专有名词、产品名、团队名在语音中的召回率。

建议字段：

- `phrase`
- `locale`
- `weight`
- `scene`
- `aliases`

示例见：

- `design/samples/asr-bias-phrases.sample.jsonl`

## 9. 用户词库

用户词库存储内容：

- 用户主动添加短语
- 系统从选词行为中学习的短语
- 用户英文高频词
- 用户拒绝润色或常用表达偏好

建议结构：

- Rime `userdb` 负责候选学习
- App 层额外维护用户自定义短语与删除记录
- 记录 `term`、`reading`、`weight`、`lastUsedAt`、`source`
- 支持软删除、撤销学习与导出/清理

示例见：

- `design/samples/user-phrases.sample.jsonl`

## 10. 词库更新机制

### 10.1 发布物

建议将词库拆成独立版本包：

- 基础 Rime 方案包随应用发版
- 热更新字典包由服务端下发清单
- 行业包按需下载
- 必要时同步下发新的 `.custom.yaml` patch

### 10.2 更新策略

- 启动后后台检查
- 仅在 Wi-Fi 或用户允许蜂窝网络时更新
- 使用 checksum 校验
- 下载完成后原子切换
- 保留上一版本回滚
- 新包落盘后，后台触发一次 Rime deploy 或增量重载

## 11. 方案与词库生成流水线

建议构建流程：

1. 维护 TSV/CSV/内部词表源数据
2. 做清洗、去重、词频归一化、敏感词过滤
3. 生成 `*.dict.yaml`
4. 合成 `*.schema.yaml` 与 `*.custom.yaml`
5. 使用 Rime deploy 工具链预编译为 `build/` 产物
6. 打包进 `resources/base/rawfile/rime/shared`

这条流水线应尽量放在离线构建或 CI 中完成，而不是在客户端首次启动时全量生成。

## 12. 排序相关特征

每条词条除了原始词频，还应有以下附加特征：

- `baseWeight`
- `userWeight`
- `recencyScore`
- `domainBoost`
- `voiceBoost`
- `llmKeepPriority`

其中 `llmKeepPriority` 用于保证在润色或纠错时，关键实体词不被轻易改写。

## 13. 行业词库

建议支持如下行业包：

- 通用办公
- 科技研发
- 教育培训
- 法务合同
- 医疗咨询

每个行业包包含：

- 专有名词
- 常用短语
- 缩写映射
- 语音热词
- 润色保留词

## 14. 词库治理

需要建立词库治理流程：

1. 词源采集
2. 清洗与去重
3. 词频归一化
4. 敏感词过滤
5. 质量评估
6. 编译打包
7. 灰度下发

## 15. 质量指标

| 指标 | 说明 |
| --- | --- |
| OOV 率 | 未登录词比例 |
| Top1 命中率 | 第一候选正确率 |
| Top10 覆盖率 | 前十候选覆盖 |
| 用户撤销率 | 上屏后马上删改比例 |
| 热词收益 | 热词包启用前后召回差异 |

## 16. 风险控制

- 用户词库不能无限膨胀，需设置容量与衰减策略
- 热词包不能污染基础排序，应有权重上限
- 敏感词、辱骂词、低质量口语需进入 blocklist 或低权重池
- 直接分发第三方 Rime 方案和词库前，必须完成 license 复核
- 客户端首次 deploy 时间必须被控制，避免首屏卡顿
