# 开发计划

## 1. 目标

基于当前 `design` 目录中的方案文档，制定一份可直接开工的实现计划，目标是按“先输入法骨架、再 `librime` 集成、再本地输入、再语音、再 LLM、最后联调与验收”的顺序推进，避免一开始就被语音和云端能力拖慢主线。

这份计划默认适用于：

- 1 名主开发 + 1 名可选后端/测试协作
- 单仓迭代
- HarmonyOS 客户端与 LLM 后端并行推进

如果是单人开发，建议按 10 到 12 周排期；如果是 2 到 3 人小团队，可压缩到 6 到 8 周。

## 2. 开发原则

1. 先把“可用输入法”做出来，再做“智能输入法”。
2. 所有智能能力都必须能被关闭，不影响基础输入。
3. 每个阶段都要有可运行、可验证、可回退的交付物。
4. 客户端和服务端接口先定稿，再进入联调。
5. 密码、安全模式、验证码等敏感路径优先实现硬拦截。
6. 中文拼音优先复用 `librime`，不在 ArkTS 侧自研完整拼音算法。

## 3. 建议开发顺序

### 3.1 主线顺序

1. IME Extension 注册与输入会话跑通
2. `librime` Native Bridge 与数据目录跑通
3. 键盘面板与候选栏骨架
4. 基于 `librime` 的中文拼音输入
5. 英文本地输入、用户词库、配置中心、安全策略
6. 设置页与权限申请
7. 语音输入
8. LLM 后端 MVP
9. 客户端润色集成
10. 联调、性能优化、测试验收

### 3.2 不建议的顺序

- 一开始先写 LLM 接口层
- 一开始就做双拼、9 键、行业词库云同步
- 在 `librime` 集成还没稳定时就开始做复杂 schema patch 与自动润色替换

## 4. 里程碑规划

## M0：工程初始化与技术基线

建议周期：`3 ~ 5` 个工作日

目标：

- 让当前模板工程具备输入法开发的最小骨架
- 明确目录结构、配置文件、日志与调试方式

交付物：

- `entry/src/main/module.json5` 引入 `inputMethod` extension 配置
- 新建 `ime/ability`、`ime/session`、`ime/panel`、`ime/config`、`ime/data`、`cpp/rime` 目录
- `OffhandInputMethodExtensionAbility` 入口文件
- `SessionManager` 空实现
- `KeyboardPanel` 占位界面
- 本地默认配置加载
- `librime` 共享数据与用户数据目录约定

验收标准：

- 工程能编译通过
- 输入法扩展 Ability 能被系统识别
- 进入输入框时能收到 `inputStart`
- 键盘面板能显示基础占位 UI

## M0.5：`librime` 编译与桥接基线

建议周期：`4 ~ 6` 个工作日

目标：

- 把 `librime` 作为 Native 依赖编入工程
- 跑通最小会话与上下文读取链路

交付物：

- `entry/src/main/cpp/CMakeLists.txt`
- `cpp/rime/RimeBridge.cpp`
- `cpp/rime/RimeSession.cpp`
- `RimeService.ts`
- `resources/base/rawfile/rime/shared` 目录与最小 schema 资源
- 初始化、deploy、createSession、processKey、getContext 的最小桥接

验收标准：

- 工程能链接 `librime`
- 首次初始化不崩溃
- 可创建一个 Rime session
- 输入简单拼音后能读到 preedit 和候选

## M1：输入法框架与安全策略

建议周期：`1` 周

目标：

- 完成输入会话主链路
- 先把“什么时候能输入、什么时候必须禁用能力”定义清楚

交付物：

- `SessionManager`
- `EditorContextAdapter`
- `SecurityPolicy`
- `ConfigRepository`
- `InputSessionContext` 数据模型
- 子类型切换逻辑

验收标准：

- `inputStart/inputStop` 生命周期稳定
- 能读取 `EditorAttribute`
- 能根据 `textInputType` 切换基础布局
- `FULL` 安全模式下自动关闭语音和润色入口

## M2：键盘 UI 与候选框架

建议周期：`1` 周

目标：

- 形成一个可交互的输入法 UI 外壳
- 候选栏、功能区、删除键、空格键逻辑可运行

交付物：

- `KeyboardPanel.ets`
- `CandidateBar.ets`
- `SubtypeSwitcher.ets`
- `KeyboardStateStore.ts`
- 基础按键模型和布局配置

验收标准：

- 中文、英文、数字布局可切换
- 删除键、空格键、回车键行为正确
- 候选栏可显示 mock 数据并响应点击
- 宿主输入框可被 `insertText/deleteBackward` 正常写入

## M3：基于 `librime` 的中文拼音 MVP

建议周期：`1.5 ~ 2` 周

目标：

- 先做稳定的中文全拼 MVP
- 让 `librime` 的 schema、候选、上屏流程闭环

交付物：

- `engine/rime/`
- `offhand_pinyin.schema.yaml`
- `offhand.base.dict.yaml`
- `CandidateAdapter`
- `RimeOptionMapper`
- 用户学习与 user_data 同步 MVP

验收标准：

- 常见高频词输入可用
- 支持完整拼音与基础简拼
- 候选点击、空格上屏、删除回退都正确
- 选词后用户学习能在下一次输入中体现

## M4：英文输入与中英混输

建议周期：`4 ~ 6` 个工作日

目标：

- 补齐英文输入
- 处理典型中英混输场景

交付物：

- `engine/english/`
- 英文词典加载
- 自动大小写
- 基础自动纠错
- 中英混输判定器

验收标准：

- 英文输入连续可用
- 邮箱、URL 场景自动倾向英文
- `HarmonyOS NEXT 输入法` 这类混输不出现明显错误拆分

## M5：用户词库、设置页与本地可观测性

建议周期：`1` 周

目标：

- 让产品具备可配置、可调试、可自我诊断的基础能力

交付物：

- `UserDictionaryStore`
- 设置页
- 个性化学习开关
- 云润色开关
- 语音开关
- `hilog` / `hiAppEvent` / `hiTraceMeter` 基础埋点

验收标准：

- 用户可关闭个性化学习
- 关闭后不再写入用户词库
- 可查看关键链路日志和时延

## M6：语音输入 MVP

建议周期：`1 ~ 1.5` 周

目标：

- 跑通本机 `CoreSpeechKit` 识别链路
- 在键盘内展示部分结果和最终结果

交付物：

- `voice/SpeechRecognizerAdapter.ts`
- `voice/VoiceSessionManager.ts`
- `VoicePanel.ets`
- 麦克风权限申请
- ASR 热词加载

验收标准：

- 可发起语音识别
- 可收到部分结果和最终结果
- 可手动取消识别
- 失败后不影响继续键入

## M7：LLM 后端 MVP

建议周期：`1 ~ 1.5` 周

目标：

- 后端具备最小润色能力
- API 契约、超时、错误码、审计链路可用

交付物：

- `POST /v1/polish`
- `GET /v1/flags`
- Prompt Registry MVP
- Model Router MVP
- 规则引擎兜底
- 鉴权与限流基础版

验收标准：

- 轻润色模式可返回稳定结果
- 超时和错误码符合设计
- 不保存原文
- 返回 `requestId`、`promptVersion`、`modelTag`

## M8：客户端润色集成

建议周期：`1` 周

目标：

- 让客户端可以调用后端润色，并以“建议”而非“强替换”方式展示

交付物：

- `polish/LlmGateway.ts`
- `polish/PolishOrchestrator.ts`
- `polish/PromptBuilder.ts`
- `PolishSuggestionBar.ets`
- 原文 / 润色建议 / 回退逻辑

验收标准：

- 手动润色可用
- 语音结束后可自动给建议
- 密码、URL、邮箱等场景自动禁用
- 用户可一键回退原文

## M9：联调、性能优化与发布准备

建议周期：`1 ~ 2` 周

目标：

- 完成系统测试、性能优化和首版发布准备

交付物：

- 端到端联调
- 性能 profiling
- 崩溃与异常收敛
- 回归测试清单
- 灰度开关
- 发布说明与已知问题列表

验收标准：

- 按键到候选刷新满足目标
- LLM 超时不阻塞输入
- 语音与润色在弱网下能平滑降级
- 核心功能通过回归清单

## 7. 代码落地顺序建议

建议后续实际编码按下面顺序提交：

1. `ime/ability`
2. `ime/session`
3. `ime/panel`
4. `ime/config`
5. `ime/data`
6. `ime/engine/pinyin`
7. `ime/engine/english`
8. `app/settings`
9. `ime/voice`
10. `ime/polish`

这样可以保证每一步都能编译、运行、验证，不会出现大块功能同时失稳。

## 8. 测试与验收门禁

每个里程碑结束前至少满足：

- 单元测试通过
- 核心路径手测通过
- 敏感场景拦截验证通过
- 无阻断级崩溃
- 性能指标不显著退化

建议门禁：

- M0-M2：以构建通过 + 手测为主
- M3-M5：补充引擎与配置单测
- M6-M8：补充语音和网络 mock 集成测试
- M9：执行完整回归

## 9. Definition of Done

一个模块算完成，至少满足：

1. 代码进入既定目录
2. 接口与数据结构有注释或配套文档
3. 关键路径有测试或验证记录
4. 异常和降级路径被覆盖
5. 日志与埋点齐全
6. 不突破隐私与安全约束

## 10. 当前待确认事项

开始开发前，建议尽快确认以下决策：

1. 输入法 profile 的最终 `metadata.name` 键名，以目标 SDK 官方文档或真机验证为准。
2. 语音输入首版是否只做在线识别。
3. `librime` 首版锁定哪个稳定 release 线，并采用源码编译还是预编译二进制。
4. 中文方案首版是否直接继承 `luna_pinyin_simp`，还是完全自维护 `offhand_pinyin`。
5. LLM 后端首版是走自托管模型，还是先接入外部网关。
6. 用户词库是否首版就做本地加密。
7. 首版是否需要设置页里的“正式化 / 轻润色 / 清晰表达”快捷切换。

## 11. 推荐立即开工项

如果现在就开始编码，建议从以下任务切入：

1. 按 [02-鸿蒙IME工程架构.md](./02-鸿蒙IME工程架构.md) 完成 `inputMethod` extension 骨架。
2. 把 `ime/ability`、`ime/session`、`ime/panel`、`cpp/rime` 目录搭起来。
3. 先做 `SessionManager + KeyboardPanel + SecurityPolicy + RimeService` 这四个核心文件。
4. 在能收到 `inputStart` 且能显示键盘占位 UI 之后，先打通 `librime` 最小 session，再进入中文拼音能力扩展。

这会是后续所有开发的稳定起点。
